# 墨境棋局 - 开发流程规范

> 本文档定义了整个项目的开发流水线，所有开发工作必须遵循此流程。

---

## 一、开发流水线总览

```
开发文档 → 需求拆解 → 模块开发 → 单元测试 → 开发日志
```

每个环节都必须有对应的文档输出，确保可追溯、可复现。

---

## 二、文档体系

### 2.1 文档结构

```
docs/
├── 00-开发流程.md          # 本文档（流程规范）
├── 01-项目概览.md          # 项目总览（单一事实来源）
├── 02-世界观设定.md        # 世界观、角色、剧情
├── 03-玩法设计.md          # 核心玩法、棋盘系统、卡牌系统
├── 04-美术规范.md          # 美术风格、素材规格、生成流程
├── 05-需求清单.md          # 待开发需求（按优先级）
└── 06-开发日志.md          # 变更历史（CHANGELOG）
```

### 2.2 文档更新规则

1. **单一事实来源**：每个信息只在一个地方定义，其他地方引用
2. **版本控制**：文档修改必须记录在开发日志
3. **需求驱动**：所有开发工作必须对应需求清单中的条目
4. **先文档后代码**：需求文档 → 代码实现 → 测试验证 → 日志记录

## 三、开发流程详解

### 3.1 需求拆解阶段

**输入**：项目概览、玩法设计、美术规范  
**输出**：需求清单（05-需求清单.md）

**流程**：
1. 从设计文档中提取功能点
2. 按模块分类（核心系统、玩法、美术、音效等）
3. 评估优先级（P0-P3）
4. 拆分可执行的开发任务
5. 记录依赖关系

**需求模板**：
```markdown
### [模块] 功能名称
- **优先级**：P0/P1/P2/P3
- **状态**：待开发/开发中/已完成/已测试
- **描述**：功能描述
- **验收标准**：可验证的完成条件
- **依赖**：依赖的其他需求
- **相关文件**：涉及的代码文件
```

### 3.2 模块开发阶段

**输入**：需求清单中的具体任务  
**输出**：代码实现

**流程**：
1. 确认需求细节和验收标准
2. 设计技术方案（如需）
3. 实现代码
4. 本地测试（手动验证基本功能）
5. 编写单元测试（如适用）

**代码规范**：
- TypeScript 严格模式
- 函数和类必须有注释
- 关键逻辑必须有单元测试
- 保持文件职责单一

### 3.3 测试验证阶段

**输入**：代码实现  
**输出**：测试记录（记录在开发日志中）

**流程**：
1. 运行 `npm run typecheck`（必须 0 error）
2. 运行 `npm run test`（必须全部通过）
3. 运行 `npm run build`（必须成功）
4. 手动测试核心流程
5. 记录测试结果

**测试命令**：
```bash
# 类型检查
npm run typecheck

# 单元测试
npm run test

# 构建
npm run build

# 运行游戏
cd dist && python -m http.server 8080
# 访问 http://localhost:8080
```

### 3.4 开发日志阶段

**输入**：完成的开发任务  
**输出**：开发日志（06-开发日志.md）

**流程**：
1. 记录变更内容
2. 记录修改的文件
3. 记录测试结果
4. 更新需求状态
5. 提交代码（如适用）

**日志模板**：
```markdown
## YYYY-MM-DD

### 新增
- [模块] 功能描述 (#需求ID)

### 修改
- [模块] 修改内容 (#需求ID)

### 修复
- [模块] 修复内容 (#需求ID)

### 测试结果
- TypeCheck: 0 error ✓
- Build: 成功 ✓
- Tests: X/Y passed ✓

### 相关文件
- src/xxx/xxx.ts
```

## 四、美术资源管线

### 4.1 美术开发流程

```
美术规范 → 生成Prompt → ComfyUI生成 → 后处理 → 应用到游戏
```

**全流程自动化**：所有步骤通过脚本执行，无需手动操作。

### 4.2 自动化管线架构

```
scripts/
└── art_pipeline.py      # 美术管线主脚本

管线流程：
1. 读取美术规范（04-美术规范.md）
2. 生成ComfyUI API请求
3. 调用ComfyUI生成素材
4. 后处理（调整尺寸、格式转换）
5. 输出到 assets/sprites/ 或 assets/backgrounds/
6. 记录生成日志
```

### 4.3 ComfyUI集成

**ComfyUI工作流**：
详见 `scripts/art_pipeline.py`，支持：
- 分层控制策略（角色一致性、动作姿态、风格统一）
- 版本管理（记录所有生成参数）
- VNCCS POSE STUDIO集成（姿态迁移）

### 4.4 美术资源规格

**角色精灵**：
- 尺寸：128×128px
- 方向：8方向（N, NE, E, SE, S, SW, W, NW）
- 动画：idle(4帧), walk(6帧), attack(4帧), run(6帧), dead(4帧)
- 格式：PNG（透明背景）

**场景背景**：
- 尺寸：800×600px
- 分层：far, mid, near, foreground
- 格式：PNG

**UI元素**：
- 卡牌：180×260px
- 按钮：120×40px（最小）
- 图标：64×64px

### 4.5 美术资源命名规范

详见 `docs/04-美术规范.md` 中的资源目录结构。

## 五、版本管理

### 5.1 版本号规范

采用语义化版本：`主版本.次版本.修订号`

- **主版本**：重大功能更新（如核心玩法重构）
- **次版本**：功能新增（如新章节、新卡牌）
- **修订号**：Bug修复、小优化

当前版本：`v0.2.4-alpha`

### 5.2 Git提交规范

```bash
# 功能开发
feat(战斗): 实现棋盘落子系统

# Bug修复
fix(存档): 修复存档加载失败的问题

# 文档更新
docs(流程): 更新开发流程文档

# 美术资源
art(主角): 生成主角精灵表
```

## 六、质量保障

### 6.1 代码质量

- TypeScript 严格模式（0 error）
- 函数注释覆盖率 > 80%
- 关键逻辑必须有单元测试
- 避免硬编码（使用常量）

### 6.2 测试覆盖

当前测试文件：
- `tests/board.test.ts` - 棋盘系统测试
- `tests/state-machine.test.ts` - 状态机测试

待补充：
- BattleEngine, MapGenerator, CardSystem 等

### 6.3 性能指标

- 目标帧率：30 FPS（移动端）/ 60 FPS（PC）
- 内存占用：< 200MB
- 首次加载：< 3秒

## 七、协作规范

### 7.1 任务分配

- 每个需求有明确的负责人
- 开发前确认需求和验收标准
- 开发中遇到问题及时沟通

### 7.2 代码审查

- 重要功能必须经过代码审查
- 审查要点：代码质量、测试覆盖、文档更新
- 审查通过后才合并代码

### 7.3 文档同步

- 代码修改必须同步更新相关文档
- 需求状态变更必须记录在开发日志
- 美术资源生成必须记录Prompt和参数

## 八、常见问题

### Q1: 如何添加新功能？
1. 在需求清单中添加新需求
2. 编写需求文档（描述、验收标准）
3. 实现代码
4. 编写单元测试
5. 更新开发日志

### Q2: 如何生成美术资源？
1. 确认需要的美术资源（角色、场景、UI）
2. 运行 `python scripts/art_pipeline.py`
3. 脚本自动生成并输出到 assets/
4. 在游戏中应用

### Q3: 如何调试游戏？
```bash
# 启动本地服务器
cd dist && python -m http.server 8080

# 访问游戏
# http://localhost:8080

# 打开浏览器开发者工具（F12）
# 查看Console日志
```
